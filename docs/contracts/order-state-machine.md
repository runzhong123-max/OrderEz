# 订单状态机契约

## 文档目的
这份文档用于统一顾客端、商家端、后端对订单状态和支付状态的理解，避免三端各自定义不同的流转规则。

本文档以根目录 [data-models.md](/E:/OrderEz/data-models.md) 中的 `Order.status` 和 `Order.paymentStatus` 为唯一状态来源。页面展示可以各自映射中文文案，但数据层状态值必须保持一致。

## 适用范围
- 顾客端小程序：用于订单列表、订单详情、确认下单后的状态展示。
- 未来商家端小程序：用于接单、备餐、出餐、完结等状态操作。
- 未来后端服务：用于订单存储、状态校验、接口返回和审计。

## 一、订单状态定义

### `pending`
- 含义：订单已创建，等待商家确认。
- 当前阶段定位：顾客提交订单成功后，订单默认进入该状态。
- 顾客端建议文案：`待商家确认`
- 商家端建议文案：`待确认`

### `accepted`
- 含义：商家已接单，确认开始处理这笔订单。
- 顾客端建议文案：`商家已接单`
- 商家端建议文案：`已接单`

### `preparing`
- 含义：订单进入制作中。
- 顾客端建议文案：`正在制作`
- 商家端建议文案：`制作中`

### `ready`
- 含义：订单已制作完成，可取餐或可上桌。
- 顾客端建议文案：`可以取餐了`
- 商家端建议文案：`待取餐`

### `completed`
- 含义：订单流程已完成。
- 当前阶段建议：堂食场景下可理解为已上桌或已完成取餐。
- 顾客端建议文案：`已完成`
- 商家端建议文案：`已完成`

### `cancelled`
- 含义：订单已取消，不再继续履约。
- 当前阶段建议：仅用于基础取消场景，不展开复杂责任划分。
- 顾客端建议文案：`已取消`
- 商家端建议文案：`已取消`

## 二、支付状态定义

### `unpaid`
- 含义：订单尚未支付。
- 当前阶段说明：顾客端本地 mock 流程可以保留该状态，但不要用中文替代状态值。

### `paid`
- 含义：订单已支付成功。
- 当前阶段说明：后续正式接入微信支付后，应由支付结果驱动该状态更新。

### `refunded`
- 含义：订单已发生退款。
- 当前阶段说明：先保留状态定义，不展开复杂退款明细和分笔退款能力。

### `failed`
- 含义：支付失败或支付未完成。
- 当前阶段说明：用于承接未来支付失败场景，当前 mock 可不主动流转到该状态，但字段必须保留。

## 三、状态流转规则

### 1. 订单状态主流程
当前阶段推荐的基础流转如下：

```text
pending -> accepted -> preparing -> ready -> completed
pending -> cancelled
accepted -> cancelled
```

### 2. 当前阶段允许的流转

| 当前状态 | 允许流转到 | 触发主体 | 说明 |
| --- | --- | --- | --- |
| `pending` | `accepted` | 商家 | 商家确认接单 |
| `pending` | `cancelled` | 顾客 / 商家 / 系统 | 当前阶段只保留基础取消，不细分责任 |
| `accepted` | `preparing` | 商家 | 开始制作 |
| `accepted` | `cancelled` | 商家 / 系统 | 仅保留基础异常取消能力 |
| `preparing` | `ready` | 商家 | 制作完成 |
| `ready` | `completed` | 商家 / 系统 | 堂食上桌或完成取餐 |

### 3. 当前阶段不允许的直接跳转
以下跳转不建议直接发生，后端应拦截：

- `pending -> preparing`
- `pending -> ready`
- `pending -> completed`
- `accepted -> ready`
- `accepted -> completed`
- `preparing -> completed`
- `ready -> accepted`
- `completed -> 任何其他状态`
- `cancelled -> 任何其他状态`

### 4. 顾客端、商家端、系统的职责边界

#### 顾客端
- 当前阶段以查看订单为主。
- 顾客可否取消订单，应严格受状态约束。
- 当前推荐规则：仅 `pending` 状态允许顾客主动取消；如果后续接入真实支付，再结合 `paymentStatus` 决定是否允许取消和是否需要退款。

#### 商家端
- 商家负责推进订单履约状态。
- 当前阶段商家可触发：`accepted`、`preparing`、`ready`、`completed`、`cancelled`。
- 商家端页面展示可以更偏操作视角，但不得发明新的底层状态值。

#### 系统
- 系统负责校验状态合法性。
- 系统可负责记录 `updatedAt`，并在未来扩展支付回调、自动取消、超时处理等能力。
- 当前阶段不实现复杂自动流转，但后端设计应保留校验入口。

## 四、支付状态流转规则

当前阶段建议的基础流转如下：

```text
unpaid -> paid
unpaid -> failed
paid -> refunded
```

### 当前阶段允许的流转

| 当前支付状态 | 允许流转到 | 触发主体 | 说明 |
| --- | --- | --- | --- |
| `unpaid` | `paid` | 系统 | 支付成功回调后更新 |
| `unpaid` | `failed` | 系统 | 支付失败或超时关闭 |
| `paid` | `refunded` | 系统 / 商家 | 未来退款成功后更新 |

### 当前阶段不允许的直接跳转
- `failed -> paid`
- `failed -> refunded`
- `refunded -> paid`
- `refunded -> unpaid`

说明：如果支付重试成功，建议创建明确的支付记录或重新发起支付流程，不要直接把旧的 `failed` 订单支付状态回写成 `paid` 而不留痕迹。当前阶段先只约束状态方向，不展开支付单模型。

## 五、订单状态与支付状态的组合建议

当前阶段建议使用以下基础组合：

| `Order.status` | `Order.paymentStatus` | 说明 |
| --- | --- | --- |
| `pending` | `unpaid` | 已提交，待商家确认，未支付或待支付 |
| `pending` | `paid` | 已提交，待商家确认，已支付 |
| `accepted` | `paid` | 商家已接单，已支付 |
| `preparing` | `paid` | 正在制作 |
| `ready` | `paid` | 可取餐或可上桌 |
| `completed` | `paid` | 已完成 |
| `cancelled` | `unpaid` | 未支付即取消 |
| `cancelled` | `refunded` | 已支付后取消并已退款 |

当前阶段不建议出现的组合：
- `completed + unpaid`
- `ready + unpaid`
- `preparing + unpaid`

如果后续业务允许“先下单后支付”，应单独更新本文件和 `data-models.md`，不要让页面或接口自行放宽解释。

## 六、页面展示映射建议

## 原则
- 数据层统一使用英文状态值。
- 页面层统一做中文映射，不要把中文状态文案写进数据库主字段或接口主字段。
- 顾客端和商家端可以展示不同中文，但底层状态必须一致。

### 顾客端建议映射

| 数据值 | 建议展示文案 |
| --- | --- |
| `pending` | 待商家确认 |
| `accepted` | 商家已接单 |
| `preparing` | 正在制作 |
| `ready` | 可以取餐了 |
| `completed` | 已完成 |
| `cancelled` | 已取消 |

| 数据值 | 建议展示文案 |
| --- | --- |
| `unpaid` | 待支付 |
| `paid` | 已支付 |
| `refunded` | 已退款 |
| `failed` | 支付未完成 |

### 商家端建议映射

| 数据值 | 建议展示文案 |
| --- | --- |
| `pending` | 待确认 |
| `accepted` | 已接单 |
| `preparing` | 制作中 |
| `ready` | 待取餐 |
| `completed` | 已完成 |
| `cancelled` | 已取消 |

## 七、当前阶段边界
- 当前先以堂食场景为主，`orderType` 先重点支持 `dine_in`。
- 当前不处理复杂退款流程，只保留 `refunded` 基础状态定义。
- 当前不处理复杂售后流程，不拆分“部分退款”“申诉”“异常完结”等状态。
- 当前只定义最小可用状态机，重点解决三端统一，不做过重设计。

## 八、对当前顾客端的直接约束
结合当前顾客端项目，后续对齐时应优先做到以下几点：

- 订单状态从当前中文值迁移为 `pending`、`accepted`、`preparing`、`ready`、`completed`、`cancelled`。
- 支付状态从当前中文值迁移为 `unpaid`、`paid`、`refunded`、`failed`。
- 页面展示中文文案放在映射层，不直接写进订单主数据。
- `createOrder`、订单列表、订单详情、订单状态展示统一以本文件为准。

## 九、变更规则
如果后续要新增订单状态、支付状态或放宽流转规则，必须按以下顺序处理：

1. 先更新 [data-models.md](/E:/OrderEz/data-models.md)
2. 再更新本文件
3. 再更新 [api-contracts.md](/E:/OrderEz/docs/contracts/api-contracts.md)
4. 最后再更新顾客端、商家端、后端代码

任何一端都不得跳过契约文档，直接新增状态值。
