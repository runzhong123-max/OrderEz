# 统一技术路径

## 文档目的
本文档用于统一当前顾客端小程序、未来商家端小程序、未来后端服务的推进方向。

它回答的核心问题是：

- 系统应该拆成哪些部分
- 为什么要拆
- 哪些文档属于三端共享基础
- 哪些文档只属于单个项目当前阶段
- 当前推荐的开发顺序是什么

本文档与根目录 [data-models.md](/E:/OrderEz/data-models.md) 配合使用：

- `data-models.md` 负责统一业务对象和字段
- 本文档负责统一项目关系、工程边界和开发路径

## 一、当前系统组成
当前推荐将系统拆成四部分：

1. 顾客端小程序
2. 商家端小程序
3. 后端服务
4. 共享文档体系

### 1. 顾客端小程序
负责顾客点餐相关流程，当前已经具备：

- 首页展示
- 菜单浏览
- 购物车管理
- 确认下单
- 本地订单生成与查看

顾客端的核心目标是：

- 让顾客顺手点单
- 让页面适合手机使用
- 不承载复杂商家维护逻辑

### 2. 商家端小程序
未来应作为独立项目存在，负责：

- 店铺信息维护
- 菜单分类维护
- 菜品维护
- 订单查看与处理
- 后续轻量运营维护能力

商家端的核心目标是：

- 低门槛
- 表单化
- 流程化
- 不做重后台

### 3. 后端服务
后端应负责：

- 店铺信息存储
- 菜单分类与菜品存储
- 订单创建与状态流转
- 用户基础信息
- 后续支付、昵称、留言等真实业务能力

后端的核心目标是：

- 承接真实业务数据
- 输出稳定接口
- 不自定义另一套字段语言

### 4. 共享文档体系
当前建议明确区分两类文档：

- 三端共享文档
- 项目独立分析文档

## 二、为什么顾客端和商家端要拆成两个独立小程序
当前建议拆成两个独立小程序，而不是继续把所有功能堆在同一个项目里，主要原因有四点。

### 1. 角色目标不同
顾客端追求的是：

- 快
- 简单
- 轻
- 一眼看懂

商家端追求的是：

- 容易维护
- 容易录入
- 容易修改
- 能处理日常经营

两者在信息密度、页面结构、流程设计上天然不同。

### 2. 工程节奏不同
顾客端更关注点餐链路和订单展示。
商家端更关注：

- 数据维护
- 状态流转
- 菜品启停
- 店铺资料管理

如果继续混在同一个小程序里，后续目录、路由、权限判断和页面复杂度都会迅速膨胀。

### 3. 真实业务更符合拆分
真实小饭店场景里：

- 顾客使用的是顾客入口
- 商家使用的是商家入口

这本身就是两个不同使用场景，不应被强行揉成一个产品壳子。

### 4. 更利于后端稳定
当前先拆清角色，后端就能围绕统一模型输出更清晰的接口：

- 顾客端接口
- 商家端接口
- 共享对象模型不变

## 三、为什么要先统一模型，再继续开发
当前最需要避免的问题不是“页面不够多”，而是三端各自发明一套字段和状态。

如果先堆功能、不先统一模型，后面大概率会出现：

- 顾客端一个字段名
- 商家端另一个字段名
- 后端再来第三种命名

这会直接导致：

- 订单结构混乱
- 金额字段不一致
- 状态流转难以落地
- 接口契约频繁返工

所以当前正确顺序应是：

1. 先统一业务对象
2. 再统一项目关系
3. 再统一状态机和接口契约
4. 最后分别推进顾客端、商家端、后端实现

## 四、当前推荐的开发顺序

### 第一阶段：统一模型与技术路径
目标：

- 固定核心业务对象
- 固定状态命名
- 固定金额字段命名
- 固定三端协作方式

### 第二阶段：统一状态机与接口契约
目标：

- 固定订单与支付状态流转
- 固定顾客端与商家端核心接口边界

### 第三阶段：稳住顾客端与模型的对齐
目标：

- 不推翻现有顾客端
- 把明显偏离统一模型的命名和状态逐步收口
- 保持下单链路可继续演进

### 第四阶段：独立启动商家端项目
目标：

- 基于统一模型启动新项目
- 先做店铺资料、分类、菜品、订单基础处理
- 不再把商家维护能力堆回顾客端项目

### 第五阶段：启动后端服务
目标：

- 把当前顾客端和未来商家端的 mock 数据结构落成真实存储
- 按统一模型设计接口
- 输出可联调契约

### 第六阶段：逐步替换本地 mock
目标：

- 顾客端先替换菜单与店铺数据
- 再替换订单创建与订单列表
- 商家端再接入维护接口

## 五、共享什么，独立什么

### 共享内容
三端必须共享：

- 核心对象定义
- 字段名称
- 状态名称
- 金额字段命名
- 订单状态与支付状态边界
- 核心接口输入输出结构

### 独立内容
三端各自独立的部分主要是：

- 页面结构
- 用户交互
- 表单流程
- 调用接口方式
- 本地状态管理细节
- 当前阶段分析结论

共享的是业务语言，不是页面长相。

## 六、当前阶段哪些先做，哪些后做

### 当前优先做
- 文档先行
- 顾客端模型对齐分析
- 商家端独立项目技术准备
- 后端契约准备

### 当前后做
- 复杂会员体系
- 复杂营销
- 配送
- 多门店
- 重型权限系统
- 复杂评论社区

## 七、当前结论
从当前阶段看，最优技术路径不是继续在一个小程序里同时推进所有角色，而是：

1. 保留当前顾客端项目，继续作为独立顾客端
2. 以 [`data-models.md`](/E:/OrderEz/data-models.md) 为共同基础
3. 新开独立商家端项目
4. 后端围绕统一模型、状态机和接口契约设计存储与接口
5. 通过共享文档而不是口头约定来维持三端一致
